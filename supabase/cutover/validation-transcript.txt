BLOCKING BUG FIXED — owned sequences were being moved twice.
  Column-owned sequences (serial / IDENTITY; pg_depend deptype 'a'/'i') follow
  their owner table AUTOMATICALLY under `ALTER TABLE ... SET SCHEMA` (per the
  PostgreSQL docs). The previous commit (6bb25c7) then ALSO ran an UNCONDITIONAL
  `ALTER SEQUENCE <old_schema>.<name> SET SCHEMA <new_schema>` for every captured
  sequence — referencing a name that no longer exists in the old schema, which
  aborts the transaction. This was introduced in 6bb25c7 and never live-tested
  (the last green live run, at f8e6447, PREDATES the exact-sequence move loop).
  Fix (cutover.sql + rollback.sql): move the tables, let owned sequences follow,
  then move a captured sequence explicitly ONLY IF it genuinely remains in the
  old schema (guarded `if exists (... relnamespace = <old>)`), and assert none
  remain in the old schema afterwards.

Also in this round:
